# MTProxy Gateway — подробная механика работы

Этот документ детально описывает, как устроен сервис `mtproxy_gateway`, как компоненты взаимодействуют между собой и как именно происходит выдача/отзыв доступа.

---

## 1) Архитектура

Сервис состоит из 3 контейнеров (docker compose):

1. **mtg** (`nineseconds/mtg:2`) — upstream MTProto proxy.
2. **gateway** (`gateway.py`) — TCP прослойка, которая слушает пользовательские порты и пересылает трафик в `mtg`.
3. **admin** (`admin_panel.py`) — веб-интерфейс для управления пользователями и наблюдения за состоянием.

SQLite БД лежит в общем volume (`/data/users.db`) и одновременно используется `admin` и `gateway`.

---

## 2) Модель доступа

В этом решении доступ к прокси определяется не токеном в трафике, а **портом пользователя**:

- каждому пользователю выделяется `listen_port`;
- gateway открывает listener на этом порту;
- при входящем TCP-подключении gateway проксирует данные в `mtg`.

Итог: `tg://proxy` ссылка работает как комбинация:

- `server=<PUBLIC_HOST>`
- `port=<listen_port>`
- `secret=<MTG_SECRET>`

---

## 3) Схема БД и её роль

### `users`

Хранит текущих пользователей:

- `id`
- `username` (unique)
- `token` (служебный, unique)
- `listen_port` (unique)
- `is_active`
- `created_at`

### `reserved_ports`

Регистрирует уже выданные порты:

- `port` (PK)
- `reserved_at`
- `reserved_by_user_id`

Назначение: **запретить переиспользование портов** после удаления пользователя, чтобы старый профиль Telegram не мог снова стать валидным для нового владельца порта.

### `gateway_control`

Канал управления gateway:

- `id=1`
- `restart_token`
- `updated_at`

Когда `restart_token` увеличивается, gateway выполняет принудительный `hard_refresh`.

---

## 4) Как работает gateway (`gateway.py`)

### 4.1 Синхронизация портов

Gateway в цикле:

1. читает активные порты (`users.is_active=1`);
2. открывает недостающие listeners;
3. закрывает лишние listeners.

Это делает метод `sync_ports()`.

### 4.2 Обработка клиента

Для каждого подключения:

1. фиксируется порт, на который пришёл клиент;
2. открывается соединение к upstream `mtg`;
3. запускается двунаправленный relay (client→upstream и upstream→client).

### 4.3 Мгновенный отзыв доступа

При закрытии порта gateway:

- закрывает listener;
- дропает **все активные подключения** на этом порту.

Это нужно, чтобы не ждать “естественного” разрыва TCP-сессии.

### 4.4 Restart-сигнал

Gateway отслеживает `gateway_control.restart_token`.

Если токен изменился, gateway выполняет `hard_refresh()`:

1. закрывает все текущие listeners;
2. завершает активные сессии;
3. заново открывает listeners только для актуальных активных пользователей.

### 4.5 TTL сессий

Параметр `client_max_age` поддерживается, но по умолчанию в compose выставлен `0` (отключено), чтобы избежать нестабильности для активных пользователей.

---

## 5) Как работает admin (`admin_panel.py`)

### 5.1 Управление пользователями

- Добавление пользователя вызывает `add_user(...)`.
- Переключение статуса меняет `is_active`.
- Удаление удаляет запись из `users`.

### 5.2 Автоматический перезапуск gateway при удалении

После удаления пользователя admin увеличивает `gateway_control.restart_token`.
Это запускает `hard_refresh` в gateway и позволяет отозвать доступ максимально быстро.

### 5.3 Ручной перезапуск gateway

Есть кнопка **«Перезапустить gateway»** (POST `/gateway/restart`) — делает то же самое через `restart_token`.

### 5.4 Мониторинг

`/monitor.json` и карточки в UI показывают:

- общее число пользователей;
- активные/отключенные;
- `upstream_ok`;
- сколько портов зарезервировано;
- сколько свободно в диапазоне;
- список активных портов.

---

## 6) Порт-аллокатор (`generate_token.py`)

При создании пользователя:

- если порт задан вручную — проверяется, что его **нет в `reserved_ports`**;
- если порт не задан — выбирается первый свободный в диапазоне, отсутствующий в `reserved_ports`;
- после успешного создания пользователя порт записывается в `reserved_ports`.

Это ключевой механизм против повторного доступа старых Telegram-профилей.

---

## 7) Жизненный цикл доступа (пошагово)

### Выдача доступа

1. Админ создаёт пользователя.
2. Пользователю назначается уникальный порт.
3. Gateway открывает порт и начинает проксирование в `mtg`.
4. Пользователь подключается по `tg://proxy` ссылке.

### Отзыв доступа

1. Админ удаляет пользователя (или отключает его).
2. Порт исчезает из набора активных.
3. Gateway закрывает listener и активные сессии.
4. Дополнительно срабатывает restart-сигнал (для мгновенного полного refresh).
5. Старый профиль Telegram больше не должен получать рабочее соединение.

---

## 8) Почему иногда в Telegram кажется, что «ещё подключено»

Telegram UI может кратковременно кешировать статус и отображать “подключён” до следующей сетевой проверки.

Фактическая проверка — попытка нового трафика через прокси:

- если порт закрыт и/или сессия дропнута, данные уже не проходят.

---

## 9) Практические рекомендации

1. Держите `GATEWAY_CLIENT_MAX_AGE=0`, если при TTL есть флаппинг.
2. Не уменьшайте диапазон портов слишком сильно — учитывайте политику `reserved_ports`.
3. Регулярно смотрите `docker compose logs -f gateway` для контроля `Port enabled/disabled` и `restart signal applied`.
4. Ограничьте доступ к web-admin (VPN/reverse-proxy auth/firewall).

---

## 10) Краткий чек-лист диагностики

1. `docker compose ps` — все 3 контейнера живы.
2. `docker compose logs -f mtg` — upstream без ошибок.
3. `docker compose logs -f gateway` — есть нужные `Port enabled` для активных пользователей.
4. `GET /monitor.json` — отображает корректные счётчики.
5. После удаления пользователя:
   - в логах gateway есть `Port disabled ...`;
   - при необходимости `Gateway restart signal applied ...`;
   - новое соединение по старому порту не устанавливается.
